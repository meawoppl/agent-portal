# Changelog

## 2.8.43

- **`handle_claude_output` reads the session row once per proxy output frame instead of twice.** The owner-only `select(sessions::user_id)` is hoisted above both consumers (image extraction and the message-insert gate); the full `Session` load — used only for `user_id` — is gone. Failure semantics per consumer are identical to before (lookup failure skips extraction and insert but never blocks `last_activity` or `OutputAck`); the second pool *checkout* is kept deliberately so no connection is held across CPU-bound base64 work and a transient pool error can't poison the ack path.

## 2.8.42

- **One `ConfirmModal` component replaces five copy-pasted confirm dialogs.** Admin user actions, settings sessions/tokens panels, and the dashboard's Delete/Leave session modals all hand-rolled the same overlay + message + cancel/confirm markup. The admin ban dialog (text-input modal) and token-renewed dialog (single "Done" button) were deliberately not migrated — they aren't confirm modals.

## 2.8.41

Two deliberate consistency changes: admin endpoints honor the dev-mode bypass like every other handler (grants nothing — the seeded dev user has `is_admin = FALSE` by column default, so dev-mode admin requests still get 403 unless manually promoted), and a stale cookie for a deleted user now yields 401 instead of a 404, matching all other authed handlers.

## 2.8.40

- **Hand-rolled base64url codecs replaced by the `base64` crate (×2, ~110 lines).** `shared/src/proxy_tokens.rs` and `proxy/src/util.rs` each reimplemented base64url; both now use a const lenient engine (URL_SAFE, no encode padding, `DecodePaddingMode::Indifferent`, allow-trailing-bits) that preserves every leniency the old decoders had: padding accepted anywhere it was before, whitespace pre-filtered, non-canonical trailing bits masked. One intentionally stricter case: length ≡ 1 (mod 4) inputs (which no encoder produces, and which the old code decoded into a garbage byte) now fail softly. Byte-identity pinned by RFC 4648 fixtures, an all-256-values round-trip, and a hardcoded `ProxyInitConfig` fixture. `base64` added to shared (WASM-safe).

## 2.8.4

- **Performance metrics: make `turn_metrics` a durable per-user archive (fix "only ~2 days show").** The Performance page only ever displayed as many days as the user's *sessions* happened to survive. Root cause: `turn_metrics.session_id` was `NOT NULL REFERENCES sessions(id) ON DELETE CASCADE`, and the aggregation/recent queries attributed ownership via `INNER JOIN sessions ... WHERE sessions.user_id = $me`. So when the hourly session-age cleanup (`SESSION_MAX_AGE_DAYS`) — or any explicit/launcher session deletion — removed a session, its metrics were cascade-deleted *and* would've been dropped by the join anyway. This directly contradicted the table's documented intent ("stick around indefinitely for long-horizon trend analysis"). New migration `2026-06-04-120000_decouple_turn_metrics_from_sessions`: adds `user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE` (backfilled from each row's session owner), makes `session_id` nullable and switches its FK to `ON DELETE SET NULL`, and adds a `(user_id, started_at DESC)` covering index. The insert path (`handle_turn_metrics_report`) now resolves and stores the session's owner on each row; the aggregation, stop-reason, and recent-turns queries filter by `tm.user_id` directly (no `sessions` join), so a row survives — and still attributes correctly — after its session is deleted. `TurnMetric.session_id` is now `Option<Uuid>`; the wire `TurnMetrics.session_id` stays non-null (`unwrap_or_default()` for orphaned rows). Migration verified reversible via `diesel migration redo`; backend builds, clippy clean, all `turn_metrics` tests pass.

## 2.6.25

- **Service unit always puts `$HOME/.local/bin` on PATH, and self-heals on update.** The launcher resolves the agent binary (`claude`) purely via PATH (`Command::new("claude")`), so the installed systemd unit / launchd plist must carry a PATH that includes it. The native Claude installer drops `claude` in `~/.local/bin`, which is *not* on the minimal PATH a systemd `--user` service inherits — so after a Claude update the daemon would spawn sessions that fail with "claude not installed." Two fixes in `launcher/src/service.rs`: (1) `generate_unit` / `generate_plist` now run their PATH through a new `path_with_local_bin` helper that prepends `$HOME/.local/bin` if absent, instead of blindly snapshotting the install-time `$PATH`; (2) a new `service::sync()` regenerates the installed unit/plist from the current executable + PATH (rewriting + `daemon-reload` only when the content actually changed) so units written by older launchers — which had no `Environment=PATH` at all — repair themselves. `sync()` is invoked before the service restart in both update paths: the dashboard-triggered `UpdateAndRestart` handler (`connection.rs`) and the `agent-portal update` CLI command (`main.rs`). Previously the auto-updater only swapped the binary and never touched the unit file, so a stale unit persisted indefinitely across updates. Four unit tests pin `path_with_local_bin` (prepend-when-absent, leave-untouched-when-present, empty-PATH, no-HOME).

- **Drop OpenSSL runtime dependency from the shipped `claude-portal` and `agent-portal` binaries — switch TLS to rustls across all four consumers.** Reported by a user on Ubuntu 20.04: the GitHub release binaries crashed immediately with `libssl.so.3: cannot open shared object file` because the prebuilt artifacts were linked against OpenSSL 3 (the build host's default) and Ubuntu 20.04 ships OpenSSL 1.1 only. The reqwest `default-features = false` removes `charset` (non-UTF-8 response decoding via `encoding_rs`) and `http2` from the build — both call sites only consume JSON HTTP/1.1 responses, so the loss is invisible at runtime. Post-fix verification: `cargo tree -p claude-portal -p agent-portal --target x86_64-unknown-linux-gnu -i openssl-sys` returns no matches; `ldd target/release/{claude-portal,agent-portal} | grep -i ssl` returns nothing. Closes the Ubuntu 20.04 crash without requiring users to rebuild against OpenSSL 1.1 or pin an older runtime — the released binaries now have **zero** OpenSSL link dependency and run on any system regardless of installed OpenSSL version.

This PR collapses the three predicates into one `match serde_json::from_str::<ClaudeMessage>(json)` dispatch inside `classify` itself: the variant is decoded once, the `ClaudeMessage::User(msg)` arm decides between the Assistant (tool-result envelope) and User (plain-text) categories on the already-parsed value via two new pure helpers `user_is_tool_result_envelope(&UserMessage) -> bool` and `user_is_plain_text(&UserMessage) -> bool` (taking the parsed struct, not the JSON string), and the `CodexEvent` parse is gated behind the `agent_type == Codex` check so Claude-only sessions never run it. The user-tool-result-before-plain-text invariant is now expressed structurally — both helpers are called in sequence inside the single `User` arm, so a future refactor can't re-order them without also changing the arm shape. The three free predicates and their docblocks are deleted, replaced by a single docblock on `classify` that enumerates the variant ordering and names the regression target (#758) for each rung. Three existing tests that asserted directly against the removed predicates (`user_tool_result_classifies_with_assistant`, `user_tool_result_with_top_level_content_still_groups`, `real_user_text_does_not_group_with_assistant`) are rewritten to assert against `classify_category(&json)` instead — the externally-observable category is the actual contract. New `classifier_exhaustive_over_realistic_messages` parametrized test feeds one canonical wire shape per recognized variant (assistant tool_use, user tool_result envelope, plain-text user, portal, codex item.started, system, result, error, unparseable garbage) at the classifier under `AgentType::Codex` (the strictly-larger surface — Claude shapes classify identically on both agent types) and asserts the right `GroupCategory` (or `None`) fires for each — so a new variant landing in `ClaudeMessage` or `CodexEvent` forces an explicit decision about which category it joins rather than silently falling through to `None`.

- **File upload: byte-cap per chunk + duplicate-index rejection (closes #785).** Static review surfaced two real bugs in the chunked file-upload path on `backend/src/handlers/websocket/web_client_socket.rs::handle_file_upload_chunk` / `handle_file_upload_start`. **Bug 1: server-side budget bypass.** The old `MAX_TOTAL_CHUNKS = 51_200` constant assumed 1 KB chunks (i.e. a 50 MB total budget), but the server never enforced any per-chunk *byte* size — `FileUploadChunk.data` is base64-encoded bytes on the wire, and the server forwarded each chunk to the proxy unchecked. A client sending 10 MB chunks against the chunk-count cap got `10 MB * 51_200 = 512 GB` of effective per-upload budget; with the `total_size: u64` declared field unchecked against the running total, the only real limit was the WS frame size. **Bug 2: duplicate-chunk early completion.** `received_count: u32` was incremented on *every* `FileUploadChunk` arrival, not on first arrival of a distinct `chunk_index` — so a client sending the same `chunk_index` `total_chunks` times tripped the `received_count >= total_chunks` completion branch, the upload was reported done, and the proxy received `total_chunks` copies of the same chunk with the other indices missing (garbage payload). Fix replaces `received_count: u32` with `received_indices: HashSet<u32>` and tracks `received_bytes: u64` alongside; chunk arrival now goes through a new pure `validate_and_record_chunk(upload, chunk_index, decoded_len) -> ChunkOutcome` step that returns `Ignore` for duplicate-index / out-of-range chunks (no counter bump, no forwarding), `Abort` with a static reason string for the byte-cap violations (drops the `PendingUpload` entry so subsequent chunks for the same `upload_id` are rejected as unknown), and `Accept` / `AcceptAndComplete` for the happy path — with `AcceptAndComplete` gated on `received_indices.len() == total_chunks` (unique-index completion, not arrival-count completion). New `MAX_CHUNK_BYTES = 64 * 1024` constant caps the decoded length of any single chunk; new `effective_max_total_bytes` field on `PendingUpload` is derived from `PORTAL_MAX_IMAGE_MB` at `Start` time and the binding cap on running decoded bytes is `min(total_size, effective_max_total_bytes)` — a smaller declared `total_size` doesn't let you exceed the server cap, and the server cap doesn't let you exceed an honestly-declared `total_size`. `Start` itself now rejects uploads where `total_size > effective_max_total_bytes` so over-large uploads die at handshake time instead of mid-stream, and base64 decode happens upfront on each chunk so a decode failure aborts the upload before the proxy sees garbage. `MAX_TOTAL_CHUNKS` is retained but raised to `65_536` (`MAX_TOTAL_CHUNKS * MAX_CHUNK_BYTES` is now ~3.2 GiB upper bound, but the byte caps are the binding limit so this constant is just a sanity guard against obviously-pathological `total_chunks` values). Wire shape unchanged. ## 2.5.73

- **Real auth-bypass fix: require `auth_token` validation on existing-session proxy reattach (closes #780).** Static review caught that `backend/src/handlers/websocket/registration.rs::register_or_update_session` reactivated *any* row matched by `params.claude_session_id` without verifying ownership — the existing-session branch only did the `verify_and_get_user` token resolution for the *new-session* path inside `create_new_session`, so a client that knew or guessed another session's UUID could attach as that proxy via `ProxyToServer::Register`, then read user input flowing into the session, forge `ClaudeOutput` / `SequencedOutput`, persist messages under the victim's `user_id`, and even mark unrelated sessions as `replaced` via the `replaces_session_id` path. Compounding the bug, `backend/src/handlers/websocket/proxy_socket.rs::handle_proxy_message` called `session_manager.register_session(key, tx)` *before* calling `register_or_update_session`, wiring the attacker's WebSocket into `SessionManager.sessions` (the routing table used by `send_to_session` for `SequencedInput`) prior to any DB check — so even with the auth check in place, a colliding-UUID attacker could briefly receive input messages destined for the legitimate proxy and the disconnect-cleanup path could clobber the legitimate registration's connection generation. Fix lands in three pieces:
  - **Token validation on every branch.** `register_or_update_session` now resolves `auth_token` → `user_id` (via the existing `proxy_tokens::verify_and_get_user`, with the same dev-mode `testing@testing.local` fallback the new-session path used) *before* either branch runs. A new `user_is_authorized_for_session(conn, session_id, user_id)` helper checks both the `sessions.user_id` owner column and the `session_members` sharing table (since historical owner rows aren't always mirrored into `session_members`), and the existing-session branch gates the reactivation update on that check. The `replaces_session_id` path is gated on the same check — without it, a valid-token user could mark *any* session as `replaced` purely by attaching `replaces_session_id` to their own registration. On authorization failure the response shape is identical to the not-found case (`SESSION_NOT_FOUND_ERROR = "Session not found or not authorized"`) so a UUID-guess probe can't distinguish "row doesn't exist" from "row exists but you don't own it", avoiding a session-ID oracle.
  - **Defer socket registration until after DB auth.** `handle_proxy_message`'s `Register` arm no longer calls `session_manager.register_session` up front; it only registers (and binds `session_key` / `connection_gen` / `db_session_id` for the cleanup path) on `result.success`. The send-side `tx` channel is unaffected by this reordering, so the `RegisterAck { success: false }` still flushes to the failing client through the same `send_task` that handles success-path replies.
  - **Clean close on auth failure.** `handle_proxy_message` now returns `ControlFlow<()>`; the `Register` arm returns `ControlFlow::Break(())` on failure, and the WS loop breaks out so no further `ProxyToServer` messages are accepted on a session that never authenticated. The cleanup path then `drop`s the owned `tx` handle and `await`s the `send_task` to natural completion, ensuring the failure `RegisterAck` flushes to the wire before the WebSocket closes (previously `send_task.abort()` could race the ack — fine for the success path because proxies keep the connection alive, race-prone for the immediate-close path this PR adds).
  - Two unit tests in `mod tests` of `registration.rs` pin the externally-observable contract: `unauthorized_reattach_error_shape_matches_not_found` asserts the unauthorized-reattach error is the same `SESSION_NOT_FOUND_ERROR` constant the not-found path returns (so the oracle stays closed even if a future refactor parametrizes either branch's error message), and `session_not_found_error_does_not_leak_existence` asserts the string contains none of `"token"`, `"forbid"`, `"auth"` to keep any well-meaning future edit from accidentally re-exposing existence info. End-to-end DB-backed coverage for the wrong-token / right-token reattach cases would require a Postgres test harness that doesn't yet exist in this crate; the helper extraction (`user_is_authorized_for_session`) keeps the authorization logic small and reviewable for the next person who adds that harness.
